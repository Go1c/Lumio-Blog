渲染优化实战

渲染 性能 中级
Lumio 2026-05-28 15 分钟 2.1k 阅读

渲染是大多数游戏帧时间的主要消耗来源。本文从工程视角出发,梳理一套从定位瓶颈到落地优化的完整思路:先用工具找到真正的瓶颈,再针对 CPU 提交、GPU 填充与带宽分别下手,最后用稳定的指标守住成果。

核心目标:在保证画面效果的前提下,尽可能降低 CPU 与 GPU 的负担,提升帧率与稳定性。优化永远要以 Profiler 数据为依据,而非凭直觉。

1. 优化思路总览

优化的第一步永远是测量。在动手之前,先用 Profiler 区分这一帧到底卡在 CPU 还是 GPU:如果 CPU 主线程时间远高于 GPU,说明瓶颈在提交与逻辑;反之则在填充与带宽。盲目优化往往事倍功半,甚至引入新的问题。

常见瓶颈大致可以归为三类:Draw Call 过多导致 CPU 提交开销高、Overdraw 严重导致 GPU 填充率不足、以及纹理与带宽引起的显存压力。后续章节会逐一拆解。

2. 减少 Draw Call

每一次 Draw Call 都伴随驱动层的状态校验与提交开销。减少 Draw Call 最有效的手段是合批(Batching):让使用相同材质的物体共享一次绘制。静态合批适合不动的场景物件,动态合批适合小网格,而 GPU Instancing 则适合大量同一网格的实例。

实践中最容易被忽略的是材质实例化:运行时修改 material 会断开合批,应当尽量复用 sharedMaterial。

// 合批友好:复用 sharedMaterial,避免运行时实例化
foreach (Renderer r in renderers)
{
    r.sharedMaterial = batchedMaterial;
}

// 大量同种网格优先使用 GPU Instancing
Graphics.DrawMeshInstanced(mesh, 0, mat, matrices);

3. 批处理与合批

合批并非越多越好。过大的合批会牺牲视锥剔除与遮挡剔除的粒度,反而把本可以剔除的几何体也提交给 GPU。合理的做法是按空间区域与可见性分组,让每个批次既足够大以摊薄提交成本,又足够小以保持剔除收益。

4. 纹理优化

纹理是显存与带宽的大头。优先使用压缩格式(移动端 ASTC、桌面端 BC7),开启 Mipmap 以降低远处采样的带宽与闪烁,并把同一物体的多张贴图打包进图集或通道复用,减少采样次数。

5. Shader 优化

片元着色器的成本会随屏幕像素数被放大,因此应把计算尽量上移到顶点阶段或预计算到贴图中。避免在片元里做分支与高次幂运算,并为移动端提供降级的 LOD 变体。

6. 实例化渲染

对于植被、碎石、子弹等海量重复对象,GPU Instancing 能用一次 Draw Call 绘制成千上万个实例,差异通过每实例数据传入。配合视锥剔除与距离 LOD,可在保证密度的同时把提交成本压到最低。

7. 结语

渲染优化没有银弹,关键在于测量 → 定位 → 验证的闭环。每做一处改动,都用 Profiler 复测确认收益,并建立基准帧用于回归。把这套流程固化下来,性能就能在长期迭代中保持稳定。
